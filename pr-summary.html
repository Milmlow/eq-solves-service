The review — three independent perspectives

Three reviewers were spun up in parallel, each holding to a different lens. They didn't see each other's work. The convergence between them is the signal.

CEO / Head of Construction
Does this jump out at scale?
Verdict: materially differentiated on compliance depth, credible as the deliverable to a Jemena or Equinix compliance officer today. Real moats already built: proper multi-tenant data isolation, the Maximo Delta importer, and engineering discipline that looks like a 50-engineer shop from a one-person team.

Three gaps before this is salesable externally: offline-first for techs, integration test coverage, and a background-job queue for report generation.
UI / UX engineering
Is the technology invisible?
Score: 7/10. Sign-in chooser with sample PDFs and a demo one-click is best-in-class. But the core technician interaction — pass/fail/N/A buttons on a maintenance check — was rendered at ~28px on phones. Below the WCAG 44px minimum, a glove-tap fail in PPE in a plant room.

Plus drift from the canonical patterns: hardcoded pill colours in 30 files, gradient violation on the onboarding wizard header, sidebar abstract labels with no first-visit nudge.
Senior software engineer
Impressive under the hood?
Verdict: mildly impressed, with caveats. Disciplined single-author codebase punching well above typical solo SaaS. Strong server-action pattern adherence. The auth surface is genuinely good (paragraph-length docstrings citing real incidents and the fix).

Biggest weakness: the type story stopped at TypeScript strict mode but never reached the database — Supabase queries were untyped. Schema drift could silently ship.
All three converged on
(1) mobile / offline is the practical risk for techs on-site  ·  (2) zero integration tests on mutations is the largest correctness gap  ·  (3) documented intent and the actual code have drifted apart in several places.

Three reviewers agreeing without seeing each other's work isn't coincidence — it's the priority list.

PR #128   Foundation Found 8 latent bugs

View on GitHub →

Supabase types wired through — and the typechecker earned its keep.

Why it matters
Until tonight every database query was un-type-checked — column renames or removals could silently ship. Generating the schema as TypeScript types and threading them through the four Supabase client helpers turns every .from('foo') into a build-time check.

Doing that immediately surfaced 8 real production bugs. None were causing crashes, but all were silently broken.
The 8 bugs found
6 × Customer Report read completed_by from maintenance checks — that column has never existed. Supervisor & reviewer names on the cover have always rendered as —. Behaviour preserved; flagged for a real sign-off-column follow-up.

2 × Compliance Report read primary_colour from the tenants table — that column lives on tenant_settings, not tenants. The fallback was always undefined. Now reads the right place.

Also included

  • Generated schema types in lib/supabase/database.types.ts (53 tables, 2 views, 152 KB)
  • 29 other drift sites marked with @ts-expect-error TODO(db-types) PR 2b so they stay visible and a follow-up PR can knock them off one-by-one without a giant change
  • One-shot helper script in scripts/ for re-running marker insertion after fixes

PR #129   Foundation First real RLS test

View on GitHub →

Integration test harness + the most load-bearing test: cross-tenant customer isolation.

Why it matters
The code reviewer's single most pointed finding: zero integration tests on mutations, zero cross-tenant RLS tests. For a multi-tenant platform serving Equinix and Jemena, one missing row-level-security policy is a data breach.

This PR stands up the harness so future tests just follow the pattern, plus delivers the proof: a four-assertion test that Tenant A's customers cannot leak to Tenant B.
The four assertions
1. User A reads their own customer → succeeds
2. User B tries to read User A's customer by id → row hidden
3. User B lists all customers → only Tenant B rows visible
4. User B injects a row carrying Tenant A's tenant_id → database rejects

If any of these ever fail, treat as a P0.

Safety gate

The test setup hard refuses to run against any URL that doesn't look local. Integration tests mutate the database — accidentally pointing them at production would be catastrophic, so the gate is hardcoded rather than configurable.

What this means in plain English

For SKS leadership

Tonight's work was risk reduction across three layers. The product gets visibly better for techs on-site (PR 1). The codebase now type-checks its database — and that immediately surfaced two bugs in the customer-facing report (PR 2). And there's now a way to prove the multi-tenant data isolation actually holds, which is the foundation of any enterprise sales conversation (PR 3).

For the customer-facing report

The Customer Report DOCX has been silently leaving the "supervisor" and "reviewer" fields blank because the code was reading from a database column that never existed. The fix preserves what the report has actually been doing (rendering —) and flags a follow-up to add a real sign-off field. Nothing visibly changes today.

For the Compliance Report

The fallback colour for the tenant primary colour wasn't working — it was reading from the wrong table. The report has been falling back to the hardcoded EQ sky-blue regardless of tenant. After tonight it reads the right field, so future per-tenant colour customisations will actually show up.

What's still parked

None of these are blockers for shipping the three PRs above. They're the natural sequence of follow-ups.

The bigger picture

The three independent reviewers converged on the same three priorities — mobile/offline for techs, missing integration tests, and documentation-vs-code drift. Tonight made meaningful progress on two of those three. Offline for techs on-site remains the highest-leverage product gap before this is salesable externally. The infrastructure is laid (idempotent mutations, audit logging, real-DB testing), but the client-side offline queue hasn't been built. That's the next strategic build.
