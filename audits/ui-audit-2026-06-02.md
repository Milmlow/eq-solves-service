# EQ Service ‚Äî UI / design-system audit

**Date:** 2026-06-02
**Auditor:** autonomous overnight session (Opus 4.8)
**Branch:** `claude-overnight/ds-drift`
**Scope:** EQ Service (`eq-solves-service`) UI surface vs the canonical EQ design profile.
**Reference:** canonical profile = `--eq-*` tokens from `@eq-solutions/tokens` (pinned `#v1.2.0`); rules: no gradients (one sanctioned sign-in panel), no shadows on static cards, token-driven colour, Lucide icons only, sentence-case plain-English copy.

This is an audit + a small set of **already-applied** fixes (committed on this branch). Findings are ranked; each carries the EQ-correct fix.

## Overall verdict

Service is in good shape. Token consumption is correct (pinned, not vendored), fonts are **self-hosted** via `next/font/google` (the model the other apps should copy ‚Äî Shell and Field both still pull Plus Jakarta from the Google CDN), and `StatusBadge` already exists as the canonical status pill. The remaining drift is small, localised, and mostly mechanical.

## Fixed on this branch (commit `1c23776`)

1. **RouteProgress gradient ‚Üí flat sky bar.** `components/ui/RouteProgress.tsx:118` used `linear-gradient(90deg, #0ea5e9, #3b82f6, #6366f1)` ‚Äî cyan‚Üíblue‚Üíindigo, **none of which are EQ colours** ‚Äî plus a blue glow. This is the canonical "stray gradient" on the design-system hit-list. Replaced with flat `var(--eq-sky)` and a sky-tinted glow. Honours the no-gradients rule and brings the one always-mounted global indicator on-brand.

2. **MonthGrid status borders tokenized + dead-var fix.** `components/calendar/MonthGrid.tsx:50‚Äì58` mixed raw hexes (`#15803D`, `#B91C1C`) with `var()` refs, and referenced **`--eq-sky-deep` which does not exist** in the token set (so `in_progress` silently always used its fallback). Now all four status borders are token-driven (`--eq-success-text`, `--eq-error-text`, `--eq-deep`, `--eq-sky`) + cancelled grey ‚Üí `--eq-gray-300`. (Note: coloured left-borders here are *data encoding* on calendar chips, which the profile explicitly permits ‚Äî the issue was hardcoded colour, not the border itself.)

3. **ReportDownloadDialog border token.** `components/ui/ReportDownloadDialog.tsx:65` unselected-option border `#e5e7eb` ‚Üí `var(--eq-border-color)`.

## Medium ‚Äî recommended next

### M1. `hover:shadow-md` on static cards ‚Äî use border swap, not shadow
- `app/(app)/maintenance/KanbanBoard.tsx:155`
- `app/(app)/maintenance/SiteGroupedView.tsx:579`

Both card buttons apply `hover:shadow-md` **and** `hover:border-eq-sky`. The profile is explicit: *"static cards sit flat with a 1px border; shadows are floating-UI only; motion is a colour/border swap, not a shadow."* The border swap is already present, so the shadow is redundant drift.

**Fix:** drop `hover:shadow-md` from both class strings (keep `hover:border-eq-sky`). Optionally add `hover:bg-eq-ice/40` if a fill cue is wanted. Trivial, no behaviour change.

### M2. Inline status pills ‚Üí canonical `StatusBadge`
`StatusBadge` (`components/ui/StatusBadge.tsx`) is documented as the canonical pill (tone soft/solid, size sm/md, leading dot). ~10 surfaces still hand-roll `rounded-full bg-*-50 text-*-700` spans:

Not all are *status* pills (some are count badges / icon-dot wrappers) ‚Äî triage per file. This is the long-tail of the "inline `bg-*-50 text-*-700` ‚Üí StatusBadge" migration already noted in `CLAUDE.md`. Incremental: migrate when each surface is next touched. Doing it in one sweep would also collapse the raw status-colour usage that currently bypasses the `--eq-success/warning/error` tokens.
